# 数据库

## 概要

- 文件：工作区根目录 `litnexus.db`
- 引擎：SQLite，运行时使用 WAL
- 迁移：连接时检查 schema；涉及结构变更前必须自动创建 `.db.bak` 备份
- 原则：文章事实、人工决定、AI 答案和“当前配置”不能相互覆盖

数据库是项目的历史记录，不是 `litnexus.toml` 的缓存。配置改变只能影响以后的流水线；不能因为删除一个问题或检索词而默默丢弃已有文章、人工复筛或 AI 结果。

## 当前过渡结构

当前 Mac 版本以 `articles` 为中心表，包含稳定的文章字段和一些动态列。

| 字段／结构 | 说明 |
|---|---|
| `epmc_id` | 文章主键；人工复筛 CSV 的唯一匹配键 |
| `pmid` / `doi` | 去重辅助键 |
| `source` | Europe PMC 来源代码，例如 `MED`、`PMC`、`PPR`、`AGR`、`CTX`；不是质量或纳入结论 |
| `title` / `abstract` / `journal_title` / `pub_year` 等 | 文章事实 |
| `title_zh` | AI 翻译标题 |
| `include` | 人工复筛结论：`yes` / `no` / `NULL` |
| `tags` | 人工自由标签 |
| `{qid}_ans` / `{qid}_rea` | 某个 AI 问题的答案与理由；当前的动态列实现 |
| `litnexus_questions` | 当前问题的元数据镜像，含归档、AI 开关和“仅未来文章”的 rowid 边界，供数据库自描述和跨库导入对齐 |
| `article_terms` | 文章与原始下载文件中命中的期刊／检索式文本关联；当前仍以文本而非稳定检索式 ID 记录 |

合并阶段以主键／唯一约束使用 `INSERT OR IGNORE` 类语义，避免重复入库。`include` 和 `tags` 是人工数据，不能由下载、合并、翻译或分类步骤重写。

## 人工复筛数据契约（Mac 与 Windows Core 已实现）

人工标注 CSV 导入只有 `epmc_id`、`include` 和 `tags` 有语义：

- `epmc_id` 必须是唯一匹配键。
- `include` 只允许 `yes` 或 `no`（大小写与首尾空格可规范化）；空白表示本次不改变原值。
- `tags` 是自由文本；空白同样表示本次不改变原值。
- 标题、摘要、期刊、翻译和 AI 结果即使出现在 CSV 中，也不参与回写。
- 导入必须先预检；重复 ID、缺 ID、有值但 ID 缺失、非法 `include`、缺少必要列均为阻断错误。
- 已有人工标注默认只读；只有用户明确选择覆盖并二次确认冲突数时才允许改写。

完整的示例、异常行处理和表格软件注意事项见[人工复筛与 CSV 导入](../guide/manual-review.md)。

## 问题与检索词的生命周期

### 已定行为

| 变更 | 以后怎么做 | 历史如何保存 | 统计如何避免混淆 |
|---|---|---|---|
| 新增 AI 问题 | 创建时明确选择“仅未来新文章”（默认）或“补答历史文章” | 不改变旧问题答案；仅未来模式记录当前 `rowid` 边界 | 当前问题集合中显示；范围外历史文章不计为“未分类” |
| 改写问题文本 | 语义变更默认新建独立问题并归档旧问题 | 旧答案仍与旧问题绑定 | 默认只看当前问题；归档项目保留在配置页 |
| 停用问题 | 以后不再调用 AI 回答 | 保留所有答案与理由 | 当前统计默认隐藏，历史模式可查看 |
| 永久清除问题 | 高级维护，备份与二次确认 | 才可能物理删除 | 不再展示 |
| 新增检索式 | 以后下载使用它 | 不改变旧下载或旧命中 | 当前列表与历史列表可区分 |
| 改写检索式 | 新建检索式版本 | 原检索式和命中仍保留 | 不把不同文本混成同一条 |
| 移除／改写检索式 | 以后不再下载旧文本 | 不删除已有文章与原始下载文件中的命中 | 当前版本尚无检索式归档 UI；稳定 ID／版本化是下一阶段迁移 |

“补答历史文章”可能调用大量 AI，因此永远不能在新增问题后无提示地自动执行。问题修改的轻微排版修正和真正的语义变更也应由用户在确认界面中区分；一旦影响答案解释，安全默认是新建版本。

### 为什么不能长期继续加动态列

当前新增问题会在 `articles` 上加入 `{qid}_ans` 与 `{qid}_rea`。增加列通常不需要重写已有文章内容，但补答历史文章需要 AI 成本；永久删除问题则会删除列和所有历史答案，结构变更风险更高。问题和检索式越多，`articles` 的列结构越不适合承载它们的历史。

因此，长期目标是将“当前规则”和“历史证据”分离：

```text
articles                         文章的稳定事实
manual_reviews                   人工 include / tags

question_definitions             问题的稳定身份与归档状态
question_versions                每次问题文本的版本
article_answers                  文章 × 问题版本 × AI 答案 / 理由 / 运行批次

search_queries                   检索式的稳定身份与归档状态
search_query_versions            检索式文本版本
download_runs                    下载范围、时间与配置快照
article_query_hits               文章 × 检索式版本 × 下载批次
```

这里的“归档”是默认删除语义：不再影响以后下载／分类和当前统计，但不会擦除历史。只有高级维护中的“永久清除”才会删除数据。

## 兼容迁移计划

规范化结构不会通过一次危险的整库重写完成。迁移应分阶段进行：

1. 备份当前数据库，并验证可恢复。
2. 新建上述规范化表；不立即删除 `articles` 的旧问题列。
3. 把当前 TOML 问题、动态答案列和检索渠道关联迁入第一版定义／版本记录。
4. 对迁移前后逐项核对文章数、人工复筛数、每个问题的有答案数及检索命中数。
5. 新代码优先读规范化表；旧动态列保持只读兼容一段时间。
6. 只有在用户明确进行高级维护、已有备份且校验通过时，才允许清理旧列。

这保证旧项目可继续打开，也保证统计可以解释“某个答案是在什么问题文本、什么运行批次下产生的”。

## 运维

| 能力 | 说明 |
|---|---|
| 统计 | 只读取本项目数据库；不重新下载文献或调用 AI |
| CSV 人工复筛导入 | 只按 `epmc_id` 写入 `include` / `tags`；先预检，默认不覆盖已有标注 |
| 数据库导入 | 与 CSV 复筛不同；需单独选择合并策略和问题列对齐 |
| 备份／导出 | 生成可恢复的数据库副本 |
| 清空文章数据 | 高风险操作；保留项目配置和列结构，必须二次确认 |

字段级精确定义以迁移版本和 selftest 为准；本文档描述产品不变量。实现变更时必须同步更新这里与[流水线](pipeline.md)。
